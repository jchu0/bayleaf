# W2 — Scoped, versioned authoring agents in an accessible library (+ a boundaries MD)

**Feature (one line):** Turn the shipped-but-unwired node-authoring agent into a *governed* capability — proposals become scoped, platform-versioned, human-approved library entries — headlined by a real **boundaries MD** that defines how any authoring agent is built and what it may do, made non-decorative by a machine-readable manifest + conformance test.

**User value + persona/journey.** Two personas. (1) The **platform maintainer** extending the six-agent roster (synthesizer / QC-triage / pipeline-repair / feedback / archivist / node-author, surfaced in Settings) to a 7th/8th agent — today that means hand-editing code + `.env.example` + the client-only Settings roster with no governing contract; W2 gives them a written, *enforced* scaffold so a new agent ships safely and repeatably. (2) The **Builder operator** (authoring journey C) who today opens a dead "Author a tool node" modal (static STAR mock) and gets an honest, advisory proposal they can review and accept. Journey touched: **Builder authoring seam + Settings roster** — deliberately **not** the Operate recording/golden path.

---

## Approaches

The three panelists converge on one truth — the maintainer's *PRIMARY* refined ask is the **scaffold + constraint contract (a real MD)**, not "wire the modal" — but differ on how much machinery rides beneath it. Presented smallest→largest; the recommendation merges all three.

### Approach A — Wire-and-transcribe: honest MD + one read-only endpoint + modal read-path (the minimal MVP)

- **Sketch.** Author `docs/design/authoring-agent-boundaries.md` as **honest transcription of invariants the code already enforces**, each line cited to a real `file:line`. Add one read-only `GET /api/builder/node-proposal?request=...` → `NodeProposal`, a near-verbatim clone of the repair endpoint, making the already-built `propose_node()` reachable. Swap the modal's static STAR mock for a real fetched proposal render and relabel the roster row honestly. Everything past "render a real proposal" (accept→card, confirm-gate+audit, platform-version stamp) is a labelled next slice.
- **Real files/seams.** Agent reused 100% unchanged: `src/bayleaf/node_author/agent.py:333` `propose_node`, `:321` `get_node_author_agent` (stub default `:327`), `:300` refusal→stub, `:316` except→stub, `:176` `_PROSE_SCHEMA` (`additionalProperties:False`, only `summary`/`rationale`). Output `NodeProposal` `src/bayleaf/node_author/models.py:187` (`advisory:Literal[True]:201`, **no verdict/confidence field**, `corpus_version`/`schema_version` stamped ~`:230`). Endpoint pattern: `GET /api/monitoring/signatures/{signature}/repair` `api/main.py:1440`. Modal: `frontend/src/components/BuilderModals.tsx:186` (`useState('STAR')`, `onClose`-only, "you review & accept" copy, "roster #5 · phase-2" badge). Roster row: `frontend/src/components/SettingsModelTier.tsx:53` (`wired:false, phase2:true`). Corpus: `src/bayleaf/node_author/knowledge/tool_cards.jsonl` (**11 entries, confirmed**).
- **Reuses-existing-pattern.** The six-agent `stub|claude` seam + the repair read-endpoint shape, verbatim; the MD *extends* the `docs/design/agents.md:66-83` intake checklist rather than inventing governance.
- **Pros.** Truly minimal (net new code ≈ one GET route + one fetch-and-render swap). The PRIMARY deliverable ships now because every constraint is already true in code. Closes two Track-A truthfulness gaps (dead modal, dishonest `wired:false` label) as a side effect. Guardrail-safe by construction.
- **Cons.** Bounded to the 11-card corpus — cannot onboard a genuinely new tool. "Scoped incorporation" is only the read path in MVP; accept→draft and confirm-gate+audit are deferred. Platform-version is a one-line constant, not a first-class registry stamp. The MD, if only prose, can silently drift from code over time.
- **Guardrail fit.** G1: a read-only GET returning a model with no verdict/confidence field cannot influence a gate. compose≠execute: endpoint returns metadata, never writes a `ProcessSpec`; the human-authored `script:`/`stub:` in `catalog.py:22-59` is untouched; uncatalogued→`exit 1` placeholder (`compiler.py:227-249`). G2: inherits stub-default + degrade-to-stub, $0.

### Approach B — Governed library + doc-drop importer (the ambitious full-gap build)

- **Sketch.** Build a first-class **Agent & Tool Library**: a new `api/library_store.py` cloning the pluggable-store shape, storing governed tool-card entries + agent-roster entries that ride the draft→pending_review→approved lifecycle. Add authoring endpoints mirroring `pipelines_lifecycle` (read-only `GET .../propose`, then `POST /{id}/submit`, `/{id}/approve`). Extend the stub with a deterministic `nextflow_schema.json`/`--help`/README **doc-drop parser** so the agent can propose a *genuinely new* tool. Governed by the boundaries MD.
- **Real files/seams.** Store to clone: `api/pipeline_store.py:81` `PipelineGraphStore` Protocol (jsonl `:126`/sqlite `:173`/postgres `:237`, degrade-to-jsonl `:5-7`). Lifecycle template: `api/pipeline.py` (`PipelineStatus draft|pending_review|approved:33`, monotonic `version`, `submitted/reviewed/approved_by`, `extra="forbid":47`) + `api/routers/pipelines_lifecycle.py:399` submit `require_role('reviewer','approver')`, approve `require_role('approver')`, `_require_status:349` 409 state machine. RBAC: `api/auth.py::require_role`. Retrieval swap point: the `Retriever` Protocol (corpus JSONL is the registry's natural home). Importer extends `StubNodeAuthor` (design note scopes it as "the deterministic importer, $0").
- **Reuses-existing-pattern.** "The `pipelines_lifecycle` pattern applied to the agent/tool library" — store, versioned envelope, RBAC, the six-agent seam, the retrieval Protocol all reused by name.
- **Pros.** The **only** tier that closes the full four-sub-goal gap: real approve-to-corpus scoped incorporation, an expandable library/roster, and the doc-drop importer that alone delivers real "bring your own tools." Every addition is confirm-gated, RBAC-gated, versioned, audited.
- **Cons.** Multi-day. The doc-drop parser is the genuine risk hotspot: unbounded free-text → typed `ToolCardEntry` mapped into the closed `ARTIFACT_KINDS` vocabulary, and it is exactly where an injection-style input could try to smuggle an executable command — needs its own spike + test surface. Migrating the trusted 11-card flat file into a governed store is a swap under a shipped retrieval path. Off the golden path; buys post-hackathon capability, not demo readiness.
- **Guardrail fit.** G1: `NodeProposal` cannot express a verdict, so a library entry can't carry a gate value — a design that let it would be disqualified, this one can't. compose≠execute: approve-to-library writes corpus *metadata*, never a `ProcessSpec` body; the human authors the runnable body, which must pass the drift-guard + live `-stub-run` gate. G2: whole flow runs on the deterministic stub; importer built stub-first; new store defaults to jsonl and degrades.

### Approach C — Governance-as-code: the boundaries MD as an *executable* contract (manifest + conformance harness)

- **Sketch.** Make the boundaries MD **self-enforcing**. Add a frozen `AgentManifest` record per roster agent declaring its capability envelope (identity + env seam; `advisory:Literal[True]`; `on_critical_path:Literal[False]`; `output_model`; the compose≠execute pin `authors_processspec:Literal[False]`; `corpus/schema/platform_version`; `register_role`/`approve_role`), plus a parametrized `tests/test_agent_conformance.py` that mechanically asserts the `agents.md` invariants against the *live* modules on every CI run. The MD cross-references the manifest field + conformance test that enforces each rule. A 7th agent is "in the library" iff it has a manifest that passes conformance.
- **Real files/seams.** Codifies `docs/design/agents.md` "Shared invariants 1–6" + "Intake a–g". Asserts the guarantees in `node_author/agent.py:321,300,316,176`. Generalizes the existing seed `tests/test_node_author.py` (already asserts `advisory is True`, no `verdict`/`confidence` attr, `PortSpec.known == kind in ARTIFACT_KINDS`) from one agent to the roster. Pins compose≠execute against `catalog.py:22-59` + `compiler.py:227-249`. `PLATFORM_VERSION` reads `importlib.metadata.version("bayleaf")` off `pyproject.toml:7` `0.1.0`.
- **Reuses-existing-pattern.** The repo's own **drift-guard culture** (the byte-for-byte `generate_reference_pipeline.py` + test that governs the compiler) applied to *agent governance*; extends `agents.md`.
- **Pros.** Delivers the maintainer's PRIMARY deliverable as its *headline* and makes it non-decorative (enforced, not just written). Lowest-risk in the panel — MVP is a test + a manifest of facts already true, zero production-code change, zero golden-path surface. Turns the crux tension into a CI tripwire: a future approach that let an agent author a runnable body **fails CI**. De-risks A and B rather than competing.
- **Cons.** Ships no new user-facing capability on its own — the modal stays a mock until an A-style endpoint lands. Value is entirely in the assertions being *real* (introspecting live modules), not nominal string-mirrors. May need a small uniform `MANIFEST` export per agent package to introspect cleanly. **Assumption flagged by the panelist:** that the four sibling agents expose the refusal-guard/except→stub/prose-only pattern *identically* to `node_author` — the harness's first run is what confirms-or-falsifies this.
- **Guardrail fit.** Net-positive/defensive: encodes G1 (`advisory`/`on_critical_path`/no-verdict) and compose≠execute (`authors_processspec:False`) as *tested* properties across the whole roster rather than convention. No new autonomy; deterministic gate untouched.

## Recommended approach + why — **merge C's self-enforcing contract (spine) + A's honest transcription & read-wire (the visible MVP) + B's lifecycle-backed incorporation & doc-drop importer (labelled later slices)**

Don't pick one — layer them, because they are naturally complementary rather than competing.

**Why this merge.** The maintainer explicitly refined W2 so the *headline output is the contract (a real MD)*, and Track-B's critical framing rewards "the most implementable, lowest-risk path that fits the existing architecture." That points to **A's MD-as-honest-transcription** (it can't drift into aspiration because every line is already true in code) — but A's lone weakness is that a prose MD *can* drift from code as the roster grows. **C fixes exactly that gap** by making the MD self-enforcing: the `AgentManifest` + `tests/test_agent_conformance.py` turn the boundaries doc from convention into a CI tripwire, and crucially make the single most load-bearing W2 constraint — *an authoring agent emits metadata, never a `script:`/`stub:` command body* — a mechanically-tested invariant (`authors_processspec:Literal[False]`) rather than a paragraph. That is the highest-value guardrail outcome the item can produce. **A supplies the smallest honest user-visible slice** (read-only endpoint + real proposal in the modal) that also clears two Track-A truthfulness findings. **B supplies the genuinely bigger capability** (approve-to-library scoped incorporation + the doc-drop importer that alone delivers "bring your own tools"), correctly sequenced as labelled follow-on slices that must *pass conformance* before they land — so the ambition is available without being on the critical path or the demo.

The result is a single dependency-ordered spine where each layer gates the next:

1. **Contract layer (headline, MVP):** the boundaries MD + `AgentManifest` + conformance harness + `PLATFORM_VERSION`. Ships the PRIMARY deliverable, enforced.
2. **Honest read layer (MVP slice 2):** read-only `GET /api/builder/node-proposal` (clone of `api/main.py:1440`) + wire `AuthorToolNodeModal` (`BuilderModals.tsx:186`) to render a real `NodeProposal` + relabel the `SettingsModelTier.tsx:53` roster row. Reserved-kind chips **shown, not wired**.
3. **Scoped-incorporation layer (deferred, labelled):** accept→`status="draft"` library entry riding the `PipelineGraph` draft→approve envelope + `pipelines_lifecycle` RBAC transitions; confirm-gated + audited via the `*_by` stamps; the human still authors the `ProcessSpec` before anything is runnable.
4. **Reach layer (deferred, spike):** the governed `library_store.py` (sqlite/postgres adapters) + the **doc-drop importer** (its own spike + safety tests) + roster-expansion UI.

The hard seam stays hard at every layer: the agent authors into `ToolCardEntry`/`NodeProposal`/`PortSpec` shapes only; the runnable `script:`/`stub:` lives solely in the human-curated `ProcessSpec` catalog; an uncatalogued tool compiles to the loud `exit 1` placeholder, never a fabricated command.

## Smallest first slice (MVP)

**Ship layer 1 + layer 2, defer 3–4.** Concretely:

1. `docs/design/authoring-agent-boundaries.md` — the governing contract, four sections mapping 1:1 to the maintainer's PRIMARY-deliverable enumeration, each line cited to a real seam: (a) templates authored-into (`ToolCardEntry`, `NodeProposal`/`PortSpec` `models.py`, target `ProcessSpec`/`Port` `catalog.py:22-59`); (b) Nextflow-integration rules (ports/version/locators only; human authors `script:`; uncatalogued→`exit 1` `compiler.py:237-245`; unknown kind→`reserved` via `PortSpec.known`; drift-guard + live `-stub-run` gate); (c) UI do's/don'ts (advisory, never auto-added, reserved-shown-not-wired, accept confirm-gated+audited, no false "Live" labels); (d) how any agent is made (the `agents.md:66-83` a–g checklist + `BAYLEAF_*_AGENT/_MODEL` seam + prose-only + degrade-to-stub + `advisory:Literal[True]` + off critical path + tests + `.env.example` entry + roster row). Plus the capability pins: metadata-not-commands; only `ARTIFACT_KINDS`; **no verdict/confidence** (G1); tool + platform + corpus + schema versioning.
2. `AgentManifest` record + the six roster manifests + `tests/test_agent_conformance.py` (generalize `tests/test_node_author.py`), so CI fails if any current or future agent violates an `agents.md` invariant — the mechanism that keeps the MD honest.
3. `PLATFORM_VERSION` constant (natural home: `src/bayleaf/identifiers.py`, which already exports `SCHEMA_VERSION`; sourced from `pyproject.toml:7`), stamped onto `NodeProposal` so a proposal pins **tool + corpus + schema + platform** — closing gap (b).
4. `GET /api/builder/node-proposal?request=...` → `propose_node(request)`, cloning `api/main.py:1440`; wire the modal to render it + relabel the roster row.

That set delivers the PRIMARY contract (enforced), pins the platform version, and makes the shipped agent honestly reachable — without opening the execution boundary or building a store.

## Effort

- Boundaries MD (transcription of already-true invariants): **<½ day**.
- `AgentManifest` + conformance harness + `PLATFORM_VERSION` stamp: **½–1 day** (mostly discovery + wiring; assertions cover behavior that already holds — the one uncertainty is whether the four sibling agents are pattern-identical, which the harness's first run resolves).
- Read-only endpoint + modal read-wire + roster relabel: **½ day** (mirror of an existing route + a fetch-and-render swap).
- **MVP total (layers 1+2): ½–1 day.**
- Deferred: accept→draft-card + confirm-gate + audit — **multi-day**; governed `library_store.py` + submit/approve UI + roster expansion — **multi-day**; **doc-drop importer — `spike-needed`** (the unbounded-input piece; scope its uncertainty separately, do not fold it into the rest).

## Risks & tradeoffs

- **Conformance-harness depth (the C risk):** if the harness asserts against copied strings instead of live modules it rots. Mitigation: assert over imported symbols (`get_*_agent`, the actual `_PROSE_SCHEMA` dict, `output_model` fields). This is the linchpin — the whole "non-decorative MD" value depends on it.
- **Sibling-agent uniformity (assumption):** the harness assumes triage/pipeline_repair/archivist/feedback/synthesis mirror `node_author`'s refusal-guard + except→stub + prose-only schema. Verified only for `node_author` this pass; first CI run confirms-or-falsifies (that is the point of the harness).
- **Doc-drop parser (the B risk):** unbounded free text → typed entry within the closed `ARTIFACT_KINDS` vocabulary while never emitting a command body — an injection surface. Keep it a spike with its own tests; structurally enforce unknown-kind→`reserved`.
- **Corpus migration (B):** moving the trusted 11-card `tool_cards.jsonl` behind a governed store changes the source of truth under a working retrieval path; the `Retriever` Protocol makes it clean in principle but it is unexercised against a store — consider dual-read until the importer is proven.
- **Perception:** layer 1 is infrastructure; a reviewer may want a visible feature. Counter-framing: "the contract that lets the 7th agent ship safely" — the maintainer's own words. Layer 2 gives the visible slice.

## Guardrail / ADR-0001 impact

**No conflict — verified structurally.** `NodeProposal` has no verdict/confidence field anywhere (`models.py:187-232`), `advisory` is pinned `Literal[True]`, and the live path is prose-only. The merge **strengthens** G1: the conformance harness makes "AI advisory, off the deterministic critical path" a *tested* property across the roster, and the manifest's `authors_processspec:Literal[False]` makes **compose≠execute** a CI tripwire — a future proposal to let any agent author a `ProcessSpec` body is disqualified *by construction*. G2 preserved: stub-default + degrade-to-stub + `BAYLEAF_*_AGENT` selector, all asserted by conformance; new store (if built) defaults to jsonl and degrades. G4: `NodeCitation.score` stays a heuristic, never "confidence." G6/G7: library additions ride the confirm-gated, RBAC-approver, audited `pipelines_lifecycle` transitions; nothing auto-adds. The single decision the maintainer must make is a **governance policy**, not a guardrail relaxation (see open questions).

## Golden-path impact

**None on the Operate recording/golden path.** The MVP is a doc + a test + a constant + a read-only endpoint + a modal read-swap — it cannot break the demo path. It is additive on the Builder authoring seam and *strengthens* the Track-A truthfulness posture (the roster's honest labels and the modal become real/test-backed). Keep the deferred library/importer **off** the demo path; do not surface a half-wired library on a hero screen — keep the phase-2 label honest until the accept slice lands.

## New ADR + `tasks.md` entries it would spawn

- **New ADR — "Authoring-agent boundaries + agent-manifest conformance contract."** Extends (does not replace) ADR-0001/0006/0009/0012 — `agents.md` already notes node-author "needed no new ADR"; a new one is warranted here because it introduces an *enforced* governance mechanism (manifest + conformance) **and** the `platform_version` binding. A second ADR ("Agent & Tool Library — governed, versioned registry") is spawned only when layer 3/4 lands (it formalizes the human-authored-body boundary + the metadata-not-commands store).
- **`tasks.md` slices (extending the T-046 line):** **T-046a** boundaries MD; **T-046b** `AgentManifest` + `tests/test_agent_conformance.py` + `PLATFORM_VERSION` stamp on `NodeProposal`; **T-046c** read-only `GET /api/builder/node-proposal` + modal read-wire + honest roster relabel; **T-046d** (deferred, gated behind conformance) accept→draft library entry riding `pipelines_lifecycle` + confirm-gate + audit; **T-046e** (deferred) `library_store.py` clone + roster-expansion UI; **T-046f** (**spike**) doc-drop importer + its safety tests + `.env.example` entry.

## Open questions for the maintainer

1. **Platform-version source.** `PLATFORM_VERSION` = `importlib.metadata.version("bayleaf")` off `pyproject.toml:7` `0.1.0` (today unreferenced by code), placed in `identifiers.py` beside `SCHEMA_VERSION` — or a dedicated pipeline/platform-version constant decoupled from the package version? (Recommendation: the constant in `identifiers.py`, reusing the module that already stamps `schema_version`.)
2. **Governance role.** Does registering a *new agent* (adds capability) require a stricter role than the pipeline `approver` that blesses a graph (`pipelines_lifecycle.py`), or is `approver` sufficient? The manifest's `register_role`/`approve_role` fields make this a per-agent policy — what is the default?
3. **Manifest home.** A per-agent `MANIFEST` constant exported from each package (clean, local) + a thin central aggregator (one place to read the library), mirroring how `__init__.py` re-exports each agent's surface — vs. a single central `roster.py`?
4. **Corpus migration timing (if layer 4 is greenlit).** Migrate the trusted 11-card `tool_cards.jsonl` into the governed store now, or dual-read alongside the flat file until the importer is proven?
5. **Doc-drop scope for v1 (the spike).** `nextflow_schema.json` only (structured, lowest parse/injection risk) — or also `--help`/README free-text (higher reach, higher risk)?
