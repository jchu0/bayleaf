# W1 ‚Äî Builder-composed graphs execute for real (beyond the seeded germline chain)

**Item:** W1 ‚Äî The Pipeline Builder actually builds *working* Nextflow workflows via the compiler (roadmap anchors: **T-123** open remainder, **#9** nf-core schema importer, **#11** visual builder; `tasks.md` #20 "Tests + live nextflow-run verify" is the same remainder).

**Feature (one line):** A graph an operator composes on the canvas runs for real once an **approved** version exists ‚Äî the run is driven by a compiler-emitted **run manifest** (external inputs + output‚Üímetric map) instead of germline hardcodes, so any all-catalogued graph executes honestly and every not-yet-runnable card reads placeholder, never a fabricated pass.

**User value + persona/journey:** Builder journey (B) + reviewer/approver persona. Today two things are dishonest: (1) `POST /api/pipelines/run` executes the **live canvas graph on RBAC role alone** ‚Äî an unsaved, unapproved draft runs directly (`api/routers/pipeline_run.py:173-189`); (2) a re-parameterized or non-germline graph reports **"failed" even when Nextflow returned 0**, because the driver hard-fails unless the four germline outputs exist (`scripts/run_giab_pipeline.py:139-144, 273-285`). This item makes "compose ‚Üí approve ‚Üí run" truthful and generalizes the run path within the curated catalog.

---

## Approaches

The three source approaches converge on one MVP (approval-gate the run) and diverge on the *mechanism* for generalizing execution. I present them as the real decision axis.

### Approach A ‚Äî Generalize the driver directly (minimal; "teach the driver about the graph")
- **Sketch:** Approval-gate the run; then pass `--<param>` for exactly the kinds in `required_inputs(graph)` and make the frozen-five parse conditional ‚Äî inside the driver, which branches on what the compiled graph declares.
- **Real files/seams (grounded):** run endpoint `api/routers/pipeline_run.py:173-189`; driver params `scripts/run_giab_pipeline.py:121-127`; frozen-five parse `scripts/run_giab_pipeline.py:139-144, 273-285`; `required_inputs(graph)` `src/pipeguard/nextflow/compiler.py:74-89` (confirmed: returns the external-input kinds); `_execute` passes reads/reference/panel conditionally `pipeline_run.py:261-266`.
- **Reuses:** `required_inputs`, the `_execute`+`_jobs` seam, the compiler's already-graph-driven `params{}` block (`compiler.py:326-352`, confirmed: emits `read1/read2/<ref_params>/<extra_params>` from the graph).
- **Pros:** Smallest diff; no new artifact; fastest MVP.
- **Cons:** The driver now inspects graph shape ‚Äî pushes execution-time knowledge into `scripts/`; harder to drift-guard (nothing committed to byte-check); the output‚Üímetric mapping lives detached from the tool that produces it.
- **Guardrail fit:** compose‚âÝexecute holds (all changes in `scripts/` + `api/routers/`). Good.

### Approach B ‚Äî nf-core schema importer + full onboarding (ambitious; the scalable end)
- **Sketch:** Everything in A, plus a deterministic importer that ingests an nf-core module's `nextflow_schema.json` ‚Üí proposes a `ProcessSpec`/`Port` skeleton into the catalog, so arbitrary tools become onboardable instead of hand-curated.
- **Real files/seams (grounded):** `ProcessSpec`+`Port` extension point `src/pipeguard/nextflow/catalog.py:22-59`; 7-tool catalog `catalog.py`; placeholder `exit 1` for uncatalogued cards `compiler.py:227-249`; the palette offers uncatalogued cards (NGSCheckMate) `frontend/src/components/BuilderShared.tsx`.
- **Reuses:** the frozen `ProcessSpec` seam ("add a tool = one spec, no compiler change"); the W2 authoring boundary; `_render_config` (the W4 profile hook, `compiler.py:326`).
- **Pros:** Scales to the #9 roadmap; one build unblocks W2/W4 seams.
- **Cons (honest ceiling):** Schema import yields **ports/params/pins, never a trustworthy `script:`/`stub:` body**. The moment machine-authored metadata carries a command it becomes a route to arbitrary command execution ‚Äî **disqualified under G1 / the W2 crux**. So the runnable body must stay human-authored or lifted from the trusted nf-core module. "Any card runs" is a half-truth: the importer buys *onboarding*, not *runnability*. This is multi-day and W2-coupled.
- **Guardrail fit:** OK **only** if the importer authors metadata and the command body stays human/nf-core-curated + passes the `-stub-run` gate.

### Approach C ‚Äî The compiled bundle *is* the run contract: emit a `run.json` manifest (recommended axis)
- **Sketch:** Everything in A's *behavior*, but the generality lives in the **compiler**, not the driver. `compile_graph` emits one extra file, `run.json`: `inputs` (from `required_inputs(graph)` mapped to param names via `REFERENCE_PARAM`, `catalog.py:224-228`) + `outputs` (per catalogued node: `{kind, glob, parser}`, `glob` lifted from the existing `Port.decl` `catalog.py:31-33`, `parser` a **new optional field on `Port`** naming a registered routine: `fastp|mosdepth|vcf_count|none`). The driver reads the manifest ‚Äî it never sees a graph. A per-node `runnable: catalogued|placeholder` flag falls out of the same `catalog_entry(tool) is not None` check the compiler already does, giving option-C "runnable tier" honesty for free.
- **Real files/seams (grounded):** `compile_graph` `compiler.py:106-170`; `required_inputs` `compiler.py:74-89`; `Port`/`ProcessSpec` `catalog.py:22-59`; `REFERENCE_PARAM`/`INDEXED_REFERENCE_PARAMS` `catalog.py:224-235`; the run-dir CSV contract `qc_metrics.csv` (written honestly with a blank `cluster_pf`, `scripts/run_giab_pipeline.py:220-230`); the drift guard + live `-stub-run` gate `tests/test_nextflow_compile.py:156-181` (confirmed: hardcoded to `germline_graph()`).
- **Reuses:** the `ProcessSpec` seam (parser is one more frozen field, co-located with the command that produces the artifact ‚Äî more honest than a detached driver parser); the drift guard (the manifest becomes a committed, byte-checked file); the run-dir CSV ‚Üí gate join (`api/card_readout.py`, `rules.py` ‚Äî untouched; only *how* the CSV is produced changes).
- **Pros:** Keeps the driver graph-blind ‚Üí **compose‚âÝexecute is cleaner** (declarative text from core codegen, execution stays in `scripts/`+`api/`); drift-guardable; subsumes option C; the manifest's `outputs`/`inputs` maps are exactly where **W3** (port‚Üídestination wiring) and **W4** (samplesheet vs single-sample fan-out) land ‚Äî one seam serves three wishlist items.
- **Cons:** More upfront design than A; the manifest is a new committed drift-guarded file (must be regenerated or the drift test fails ‚Äî that is the guard working); generalized output parsing is genuinely hard and honesty-sensitive (**spike-needed** for the second parser set).
- **Guardrail fit:** Strong. `run.json` is pure text from `src/pipeguard/nextflow/` (compose); it declares *what to parse*, never a verdict ‚Äî the rule engine still decides from `qc_metrics.csv` (G1 intact). A placeholder still `exit 1`s; a declared-but-missing output is still a loud fail; no gate metrics ‚Üí honest complete + missing-metric card, never a fabricated pass.

## Recommended approach + why ‚Äî **merge: MVP approval-gate (A/B/C converge) + Approach C's manifest as the generality mechanism + Approach B's honest ceiling as the boundary**

Do the **approval gate first as a standalone slice** (all three converge; it is the mandated Resolved #2), then generalize execution via **Approach C's `run.json` manifest** rather than A's driver-branching, because the manifest keeps the driver graph-blind (cleaner compose‚âÝexecute), is drift-guardable through the *existing* byte-for-byte test, co-locates each output‚Üímetric parser with the `ProcessSpec` that emits it, and ‚Äî decisively ‚Äî the manifest's `inputs`/`outputs` maps are the shared spine W3 (port wiring) and W4 (per-sample fan-out) already need. Adopt **Approach B's honesty ceiling as a hard boundary, not a build**: decline the nf-core importer for MVP; when it lands, it authors metadata only and the `script:`/`stub:` body stays human/nf-core-curated behind the `-stub-run` gate. Fold in **Approach A's graph-driven `--<param>`** as the manifest's `inputs` consumer, and **Approach 1's catches**: the honest **422 pre-run** for a graph that doesn't emit the gate contract, and the **G7 audit-event gap** (a Builder-run emits no audit event today ‚Äî add one).

**Grounded correction to the source approaches:** the store helper to reuse is the existing **`last_emitted(store, name)`** (`api/pipeline_store.py:392`), which returns the *emitted/approved baseline* ‚Äî plus **`_require_status(...)`** (`pipelines_lifecycle.py:349`) for the 409 pattern. There is **no `latest_approved` helper** (Approach 3 invented that name); `latest_record` (`pipeline_store.py:339`) returns the working record of any status. Approval already runs beside the gate: `submit_pipeline` (`pipelines_lifecycle.py:400`) and `approve_pipeline` (`:425`) are built, approver-only, and `approve` calls `record_emission` to stamp the approved baseline. `PipelineGraph.status` (`api/pipeline.py:33,85`) is the state ‚Äî no new store or state machine.

## Risks & tradeoffs
- **Run-the-stored-approved-graph is a contract change.** The stored `graph` is an opaque `builder/0.1` envelope (`api/pipeline.py:27,56` ‚Äî stored as-is, never node-validated). The run modal today posts a compile-ready `NextflowGraphBody` directly (`frontend/src/components/BuilderModals.tsx:901`: `graph: {...graph, name}`), while `savePipeline` stores the opaque envelope. **Assumption to re-ground (G5):** that the saved envelope is the same node/edge shape the compile path (`_to_graph`, `pipeline_run.py:154-159`) consumes ‚Äî verify when the concurrent Builder session lands. If shapes differ, a small envelope‚Üí`CompileRequest` extractor is needed.
- **Generalized output parsing is honesty-sensitive** ‚Äî a missing metric must render blank/absent, never fabricated (mirror the existing blank-`cluster_pf` precedent, `run_giab_pipeline.py:225-229`). This is why Slice 2 is manifest-first and germline-first.
- **Slice-2 generality is bounded by fixtures.** `_catalog()` supplies only reads/reference/panel_bed (`pipeline_run.py:59-88`, `_KIND_TO_CATEGORY` = 3 kinds, `:46`). Per Resolved #1 this stays fixtures-only; the realistic reach is germline-family subgraphs / re-parameterizations (QC-only, align-only), not genuinely new external-input pipelines ‚Äî honest, less than "any graph."
- **Demo choreography, not demo risk.** The recording golden path uses pre-gated `mock_run_01`/intake, so it is untouched; but the *live* Builder beat gains Save‚ÜíSubmit‚ÜíApprove‚ÜíRun clicks ‚Äî re-choreograph, and keep generalized parsing + any freshly-onboarded tool off the 3-minute recording (label stub/config-verified, not cluster-verified).
