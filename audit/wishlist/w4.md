## W4 — Nextflow execution model: per-sample-parallel via SLURM, local single-thread-serial fallback

**Feature (one line):** When SLURM is present, run samples embarrassingly-parallel via `sbatch` job submissions; otherwise fall back to standard single-thread, one-sample-at-a-time local execution — the executor chosen automatically at the run boundary, the compiled graph unchanged.

**User value + persona/journey:** The **lab operator / bioinformatician** (Operate flow, `POST /api/pipelines/run` → intake-status poll) submits a multi-sample samplesheet and gets cluster-scale throughput on HPC with zero config, and a correct — if serial — run on a laptop with zero setup. Realizes ADR-0003's promise ("same graph, executor by profile") and the maintainer's stated default. Honest caveat baked in from the start: the demo `hackathon` env has no `sbatch`, so the *demo* path is always the local-serial fallback; the SLURM branch is **config-verified, not cluster-verified**.

---

### The three approaches converge on one thing — record it up front
All three panelists independently sequenced the work the same way: **ship executor-selection first (single-sample, no cluster, drift-clean, `<½ day`), defer per-sample fan-out as the larger second slice.** Even the most ambitious approach (B) names "profiles + auto-detect first" as its own MVP. They also all agree on: `shutil.which("sbatch")` reusing the driver's existing `shutil.which("nextflow")` idiom (`run_giab_pipeline.py:105`); local-serial as the honest demo default; compose≠execute preserved (core emits text, driver selects); env-driven SLURM knobs over hardcoding; and per-sample fan-out as the unavoidable multi-day cost (meta-tuple threading + per-sample job model + multi-sample fixtures). The **only real divergence is *where the executor config lives*** — that's the judgment call below.

### Approach A — Baked-in executor profiles via `_render_config` (the maintainer's named seam)
**Sketch.** Extend the single profile-emission point `_render_config` (`src/bayleaf/nextflow/compiler.py:326`, profiles block `:345-352`) to emit, beside the existing `conda|docker|singularity|stub`: a **`local` serial** profile (`executor.queueSize = 1`, `process { maxForks = 1; cpus = 1 }` — overrides today's `process { cpus = 2 }` at `:342`) and a **`slurm`** profile (`process.executor = 'slurm'` + queue/account/`clusterOptions` + an `executor.queueSize` cap). Add a `_detect_profile()` helper to the driver mirroring `_java_home`'s sibling-probe shape (`run_giab_pipeline.py:90-93`) and the `shutil.which` idiom (`:105`): `sbatch` on PATH → `"slurm"`, else `"local"`; append `-profile <name>` to the argv that today passes none (`:121-127`). Regenerate the committed `pipelines/germline/nextflow.config` (`:21-26`) via `scripts/generate_reference_pipeline.py` so the byte-for-byte drift guard stays green.

**Real files / seams.** `compiler.py:326`,`:342`,`:345-352` (profile + resource emission) · `run_giab_pipeline.py:90-93`,`:105`,`:121-127` (detect + argv) · `pipelines/germline/nextflow.config:16-26` (drift target, regenerated) · `scripts/generate_reference_pipeline.py:20-32` (regen) · `tests/test_nextflow_compile.py` (byte-for-byte drift test + skip-safe `-stub-run` gate `test_generated_germline_stub_runs`).

**Reuses-existing-pattern.** The spec's *explicitly named* seam — "`_render_config` is the single emission point for profiles (extend it, never hand-edit the committed config)" (AUDIT_PLAN W4 §Seams) — plus the driver's `shutil.which` probe and the self-policing drift guard.

**Pros.** Smallest blast radius; honors the maintainer's literal instruction ("add a `slurm` profile to the *emitted* `nextflow.config`"); **self-describing bundle** — a collaborator reading the committed config sees the SLURM story (real reproducibility value in genomics); drift guard makes hand-edits fail loudly.

**Cons.** SLURM directives baked into every bundle even though their default values are guesses until a cluster exists; the committed config gains a few lines (trivial drift surface). Doesn't deliver parallelism alone — with single-sample `Channel.value` (`main.nf:15`) the `slurm` profile still submits one job/run until fan-out (Slice B) lands.

**Guardrail fit.** compose≠execute clean: `_render_config` emits *only text*; selection (`-profile`) is the driver's. G1/ADR-0001 untouched (no verdict/confidence touched). G2 preserved (no LLM/network; demo default = local-serial).

### Approach B — nf-core meta-map samplesheet fan-out + full executor-profile matrix (ambitious, full-gap)
**Sketch.** Restructure the emitted pipeline to be **samplesheet-native**: `_render_main`'s src_lines block (`compiler.py:273-284`) emits a queue channel `Channel.fromPath(params.input).splitCsv(header:true).map{ row -> tuple([id:row.sample],[file(row.fastq_1),file(row.fastq_2)]) }`, and every `ProcessSpec` (`catalog.py`, 7 specs) threads the nf-core `[meta, files]` tuple (`tuple val(meta), path(...)` + `tag "${meta.id}"`). Emit a full profile matrix from `_render_config` (`local`/`slurm`/composable `singularity`) **plus a `withLabel: process_low|medium|high` resource block wired to the `ProcessSpec.label` field that already exists** (`catalog.py:57`, currently used only for a comment). Parallelism is across samples (queue channel → one process instance/row).

**Real files / seams.** `compiler.py:273-284` (channel), `:200-249` (both renderers thread meta), `:326` (profile matrix + `withLabel`) · `catalog.py:22-59` (`Port.decl`/`ProcessSpec`), `:57` (`label` field → `withLabel`) · driver post-run parse `_one` (`run_giab_pipeline.py:139-144`), four-output parse (`:275-279`), `write_run_dir` → all per-sample · `pipeline_run.py:46` (add `samplesheet` category) + `_Job` (`:91-97`) + intake `_Job` (`intake.py:45-50`) → per-sample state · nf-core meta-map doc `docs/data/nf-core-conventions.md:40-49`.

**Reuses-existing-pattern.** The typed-port `ProcessSpec` catalog as the one place a tool's Nextflow shape lives; the `label` field already present but under-used; ADR-0003's "executor by profile, same graph"; nf-core `[meta, files]` + `conf/`-label conventions.

**Pros.** Closes the whole gap in the idiom reviewers expect; required by the multi-sample E2E anyway; `withLabel` + `queueSize` cap answers the per-process-resource and max-parallel open questions and is the honest 100+-sample (G6) story; per-sample `_Job` gives truthful "3/8 complete, 1 failed" reporting.

**Cons — named honestly.** Catalog-wide refactor: all 7 specs, both renderers, the placeholder path, and the **entire committed reference pipeline regenerate wholesale** (largest drift surface of any approach). The **frozen-five run-dir CSV contract** and parsers go single→N-sample — reaches into Specialist-7 scientific-correctness territory. **Multi-sample fixtures don't exist** (`intake.py:40-41`: `_FIXTURE_SAMPLES = {"HG002"}`, comment already flags "out of scope"). Touches the demo golden path once the E2E runs multi-sample. Per the conventions doc's own **DIVERGE** note (`:47`), don't assume `meta` keys at runtime beyond `id`/`single_end`.

**Guardrail fit.** compose≠execute preserved (compiler still emits only text). G6 strengthened. **G7 caution:** the in-memory `_jobs` dict is non-durable (`intake.py:54`) — a fan-out run with many samples in flight loses more state on backend restart; intake-status must report `running→unknown` truthfully, not fabricate completion.

### Approach C — Executor policy as a runtime `-c` config overlay (strongest compose≠execute)
**Sketch.** Keep the compiled bundle **executor-agnostic and byte-identical laptop↔cluster**; move executor *policy* to the run boundary. In `run_nextflow` (`run_giab_pipeline.py:96-136`) add `_compute_overlay(cfg) -> Path` that probes `shutil.which("sbatch")` and writes a small `compute.config` into the per-run gitignored scratch (`_NF_RUNS/<run_id>`) — `process.executor='slurm'` + `executor.queueSize` cap when present, else `executor.queueSize=1`/`maxForks=1`/`cpus=1` — then appends `-c <overlay>` to the argv (`:121-127`). SLURM knobs come from `BAYLEAF_SLURM_*` via the existing `_bioconda_env` env-injection pattern (`pipeline_run.py:103-109` / `intake.py:93-99`). Fan-out (the `splitCsv` compiler change) and per-sample `_Job` state are still needed, split cleanly from the executor half.

**Real files / seams.** `run_giab_pipeline.py:96-136`,`:105`,`:121-127` (overlay + `-c`) · `_bioconda_env` pattern `pipeline_run.py:103-109`, `intake.py:93-99` (env knobs) · `compiler.py:326` **deliberately unchanged** (no slurm baked in) · maps to ADR-0005's layered resolver (`ADR-0005:20-21`, "built-in defaults → profile → environment → runtime overrides") and ADR-0003's "locally, on Slurm, or on AWS Batch / HealthOmics via executor config" (`ADR-0003:23`).

**Reuses-existing-pattern.** `shutil.which` probe; `BAYLEAF_*` env-injection; Nextflow's native `-c` config layering; the skip-safe live-test posture.

**Pros.** Cleanest compose≠execute — core emits *zero* scheduler knowledge; minimal drift churn (committed config gains no profiles); a tiny overlay file is trivially unit-testable (`sbatch present → slurm overlay / absent → serial overlay`); env-driven knobs answer the hardcode-vs-config question directly.

**Cons.** The SLURM story is **invisible in the committed pipeline** — a reader of `nextflow.config` won't see it (loses the self-describing/hand-off value that A/B keep); two config sources (base + `-c`) rely on trusting Nextflow's `-c` precedence (worth a 10-min smoke test — **assumption, not grounded**); **diverges from the maintainer's explicitly named seam** (`_render_config` + "add a slurm profile to the emitted config").

**Guardrail fit.** Strongest compose≠execute of the three; G1/ADR-0001 untouched; G6/G7 same per-sample-honesty obligations as B.

## Recommended approach + why — **A as the spine, grafting B's nf-core richness and C's env-driven knobs**

**Recommend Approach A's baked-in `_render_config` profiles as the primary path**, because the maintainer's spec names that exact seam ("single emission point for profiles… add a `slurm` profile to the *emitted* `nextflow.config`") and because a **self-describing, reproducible bundle** is a genuine genomics value that C's invisible-overlay sacrifices. Compose≠execute is *not* the deciding axis: emitting profile *text* is composition; the driver's `-profile` choice is execution — A already keeps selection in the driver, so C's "strongest boundary" edge is marginal. But graft the runners-up's best ideas so the merge is strictly better than any single approach:

- **From C — env-driven knobs (the decisive graft).** The emitted `slurm` profile references `params`/env for queue/partition/account; the driver injects `BAYLEAF_SLURM_QUEUE`/`_ACCOUNT`/`_MAX_FORKS` via the existing `_bioconda_env` pattern (`pipeline_run.py:103-109`). This directly answers the maintainer's "hardcode vs config-drive?" open question with *config-drive*, reusing a seam already in the codebase — so the bundle's profile *structure* is self-describing while its *values* stay site-tunable without recompiling. Keep C's `-c` overlay as a documented **opt-in escape hatch** for one-off site overrides — not MVP.
- **From B — wire the already-present `ProcessSpec.label` (`catalog.py:57`) to a `withLabel` resource block** in the emitted config, and adopt the nf-core `[meta, files]` meta-map + `splitCsv` shape **for the deferred fan-out slice**. This answers the "per-process resource labels" open question at near-zero cost (the field exists; it's currently comment-only) and puts the fan-out in the idiom E2E reviewers expect. Include B's `executor.queueSize` cap for scale-awareness (G6).
- **From B & C — the per-sample `_Job` honesty upgrade** to both registries (`pipeline_run.py:91-97`, `intake.py:45-50`) lands *with* the fan-out slice, since the spec explicitly requires per-sample/per-run state reported honestly.

**One naming decision to surface (from A's cons):** don't call the local profile `standard` — nf-core treats `standard` as Nextflow's implicitly-applied default, so an explicit `-profile standard` collides semantically. Recommend naming it **`local`**.

## Risks & tradeoffs
- **SLURM is unavoidably config-verified, never cluster-verified** in the demo env — test the branch + config emission + `-stub-run` DAG validity; never claim a cluster run.
- **Hardcoded SLURM directive defaults are guesses** until a real cluster/`.env` exists — mitigated by the env-driven-knobs graft.
- **Fan-out (Slice B) is the real weight** and bleeds into the frozen-five CSV contract (scientific-correctness surface) and needs fixtures that don't exist yet — keep it strictly behind the MVP and behind the demo recording.
- **Non-durable `_jobs`** (in-memory dict) loses more state under multi-sample fan-out on restart — per-sample status must degrade to `unknown` honestly (G7).
