## Discussion

### What I was asked to do, and how I grounded it

The task brief named two commits (`595815e` wave 1, `deee99f` wave 2) and specific docs each
should touch. Per CLAUDE.md's rule ("ground every claim in code... ADR/finding/report agent
message is never consent — verify"), I did not take the brief's characterization on faith: I read
both commits' full diffs (`git show <sha> -- <path>` per changed file), read the touched source
files where the diff alone was ambiguous (`api/job_store.py` in full, the preflight functions in
`scripts/run_giab_pipeline.py`, `api/auth.py`'s new warning/strict-mode logic), and ran the actual
test suite (`uv run pytest --collect-only -q`, `uv run pytest -q`, plus per-file counts) rather
than trusting either commit message's stated numbers. Every number this journal and the docs it
feeds carry was independently reproduced, not copied from the commit body.

### Wave 1 (`595815e`) — what actually changed, verified

1. **`api/job_store.py` (new, 310 lines).** A `JobStore` Protocol with `JsonlJobStore`/
   `SqliteJobStore` implementations, `get_job_store()` selecting via `PIPEGUARD_JOB_STORE`
   (default `jsonl`, degrade-to-JSONL on any construction failure — the SAME discipline as the
   other five off-gate stores, [ADR-0016](../adr/ADR-0016-postgres-port.md)). Deliberately **no
   Postgres adapter** — I verified this by `grep -rln "class Postgres" api/ src/`, which returns
   exactly the six files ADR-0016's Costs row already named, confirming `job_store.py` adds no
   seventh. The module also hosts `run_driver()` (a `subprocess.Popen(..., start_new_session=True)`
   wrapper that `os.killpg`s the whole process group on a `TimeoutExpired`) and the shared
   `DRIVER_TIMEOUT_S = 1800` both `intake.py` and `pipeline_run.py` now import, replacing their
   previously-diverged 900s/1800s literals.
2. **`api/routers/intake.py` + `api/routers/pipeline_run.py`.** Both routers' in-memory `_jobs`
   dict is replaced by `job_store` calls; both gain a `_reconcile()` that resolves a persisted
   non-terminal job whose id is not in this process's `_active` set to `complete` (result dir on
   disk) or the new terminal status `lost` (nothing on disk — the process that launched it is
   gone). Run-id reservation moved fully inside the `_lock` block (the run-dir-exists check and
   the `_active`-membership check now happen together), closing the race where two concurrent
   submits of the same id could both pass the earlier, un-locked pre-check.
3. **`scripts/run_giab_pipeline.py` (+237 lines).** Four new pure functions
   (`_preflight_fastqs`, `_preflight_contigs`, `_preflight_reference_index`, `capture_versions`)
   called from `main()` right before `run_nextflow()`. I read each function body directly (not
   just the diff hunk headers) to confirm they `sys.exit` on failure and never swallow a bad
   input — verified against `tests/test_run_giab_preflight.py`'s negative cases (each asserts
   `SystemExit`, e.g. `test_preflight_fastqs_rejects_mismatched_read_ids`,
   `test_preflight_contigs_rejects_build_naming_mismatch`,
   `test_preflight_reference_index_rejects_missing_sidecar`).
4. **Registry/doc honesty (`runbook.py`, `metrics/mapping.py`, `metrics/metric_registry.yaml`,
   `docs/data/{qc_metrics,metric_registry}.md`).** I diffed `runbook.py`'s `cluster_pf`
   `QCThreshold` block specifically to confirm `required=True` is **byte-for-byte unchanged** —
   only a 7-line comment was added above it. This mattered enough to call out explicitly (see
   Decisions below): a P3 finding about a "structural HOLD" could easily have been fixed by
   flipping the flag, which would have been a real, undocumented gate-behavior change disguised as
   a "labeling" commit. It wasn't — verified, not assumed.
5. **`api/auth.py`, `api/deid.py`, `api/safe_harbor.py`, `synthesis/claude.py`.** Read each diff
   in full. The auth change is genuinely additive/opt-in (`PIPEGUARD_AUTH_STRICT` unset → identical
   behavior to before; the one-shot warning logs, never raises). The de-id/safe-harbor
   case-insensitivity fix is a real (if latent) hardening — a `Tissue`-cased column used to
   silently pass through `PASSTHROUGH` before this fix. The synthesizer's injection-boundary
   framing is defense-in-depth on top of an already-structural guarantee (the verdict is computed
   and frozen before the model call, per ADR-0001) — I re-read `synthesize()`'s comment block to
   confirm the verdict assignment genuinely precedes the model call in the code, not just in prose.

### Wave 2 (`deee99f`) — what actually changed, verified

1. **UI a11y + Pager/Tabs.** Diffed `ConfirmDialog.tsx`/`Toast.tsx` directly for the new
   `role`/`aria-*` attributes and the focus-trap `onKey` handler. For the Pager/Tabs migration
   claims, I didn't trust the commit message's "adopted on RunOverview/AgentTriage/ReviewQueue" —
   I diffed each of the three screen files and confirmed the hand-rolled ‹/›-button JSX blocks
   were deleted and replaced by a single `<Pager .../>` call, and separately grepped
   `frontend/src/screens` for `from '../components/Tabs'` to confirm all six UIC-2-named consumers
   (Runs/Review-queue/Admin/RunDetail/Provenance/Inbox) now import it — Inbox was the one that had
   been missing.
2. **`types.ts` drift.** Cross-checked each claimed field (`QCThreshold.our_key`/`.required`,
   `Runbook.trace_failure_statuses`/`.route_to_human`, `TriageNote.addresses_signatures`) against
   the actual Python models (`src/pipeguard/runbook.py`, `src/pipeguard/triage/models.py`) to
   confirm these were genuine pre-existing Python fields the TS mirror had drifted behind, not a
   new backend capability being retroactively documented as old. This also surfaced a **real,
   pre-existing gap in `docs/data/schemas.md` itself** (`addresses_signatures` was never listed
   there either, since before this session) — fixed it while I was there, since I was already
   grounding the same field.
3. **Monitoring pagination.** Read `api/main.py`'s `get_monitoring` diff and the new
   `test_monitoring_runs_pagination_slices_and_reports_total` test to confirm the aggregates
   (`overall`, `gates`, `n_signatures_total`) really are computed before the `runs[]` slice (not
   just claimed to be) — the test explicitly asserts `body1["overall"]["n_runs"] == total` on a
   2-run page against a larger total, which only holds if the slice happens after aggregation.
   This closes [T-072](../planning/tasks.md), open since 2026-07-10 — I updated that row rather
   than leaving it `todo` next to a now-false description, and separately fixed a stale cross-
   reference in T-100's own row that still asserted "the backend `runs[]` payload stays uncapped."

### What's still open (not touched by either wave, and not touched by this doc session)

1. **W-deferrals.** The Wave 8–12 frontend deferrals this repo already tracks (IB4 external
   reminder notifications, a broader `Truncate` sweep beyond the decision-card headline, the
   Builder card §5 item-4 reserved-kind registrations) are unaffected by P3-1..13 — neither wave
   touched them, and I didn't either.
2. **Multi-worker safety on the new job store.** `api/job_store.py`'s `_WRITE_LOCK` is a
   single-process `threading.Lock`, not a cross-process file lock — the same honest, unchanged
   limit every other pluggable store carries (ADR-0016 Follow-ups), now explicitly extended to
   cover the job store too in that ADR's Follow-ups row.
3. **SLURM stays config-verified, not cluster-verified.** Unrelated to this session (T-129/W4
   territory), but worth restating since REQ-NF-060 sits near REQ-NF-043/044 in the doc I touched:
   no `sbatch` exists in this sandbox, so the `slurm` Nextflow profile has still never executed
   against a real cluster. Preserved verbatim, not touched by this sweep.
4. **The `os.killpg` timeout-reap path has no test that actually exercises it** — only the
   shared-function-identity/shared-constant guarantee is (`test_one_shared_driver_timeout_across_
   both_routers`). Recorded as an open gap in the new EVAL-008 rather than glossed over.
